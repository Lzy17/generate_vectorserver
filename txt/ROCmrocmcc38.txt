<h1>Compiler Reference Guide</h1>
<h2>Introduction to Compiler Reference Guide</h2>
<p>ROCmCC is a Clang/LLVM-based compiler. It is optimized for high-performance
computing on AMD GPUs and CPUs and supports various heterogeneous programming
models such as HIP, OpenMP, and OpenCL.</p>
<p>ROCmCC is made available via two packages: <code>rocm-llvm</code> and <code>rocm-llvm-alt</code>.
The differences are listed in <a href="rocm-llvm-vs-alt">the table below</a>.</p>
<p>:::{table} Differences between <code>rocm-llvm</code> and <code>rocm-llvm-alt</code>
:name: rocm-llvm-vs-alt
| <strong><code>rocm-llvm</code></strong>                                     | <strong><code>rocm-llvm-alt</code></strong>                                                                                                             |
|:---------------------------------------------------:|:-------------------------------------------------------------------------------------------------------------------------------:|
| Installed by default when ROCm™ itself is installed | An optional package                                                                                                             |
| Provides an open-source compiler                    | Provides an additional closed-source compiler for users interested in additional CPU optimizations not available in <code>rocm-llvm</code> |
:::</p>
<p>For more details, see:</p>
<ul>
<li>AMD GPU usage: <a href="https://llvm.org/docs/AMDGPUUsage.html">llvm.org/docs/AMDGPUUsage.html</a></li>
<li>Releases and source: <a href="https://github.com/RadeonOpenCompute/llvm-project">https://github.com/RadeonOpenCompute/llvm-project</a></li>
</ul>
<h3>ROCm Compiler Interfaces</h3>
<p>ROCm currently provides two compiler interfaces for compiling HIP programs:</p>
<ul>
<li><code>/opt/rocm/bin/hipcc</code></li>
<li><code>/opt/rocm/bin/amdclang++</code></li>
</ul>
<p>Both leverage the same LLVM compiler technology with the AMD GCN GPU support;
however, they offer a slightly different user experience. The <code>hipcc</code> command-line
interface aims to provide a more familiar user interface to users who are
experienced in CUDA but relatively new to the ROCm/HIP development environment.
On the other hand, <code>amdclang++</code> provides a user interface identical to the clang++
compiler. It is more suitable for experienced developers who want to directly
interact with the clang compiler and gain full control of their application’s
build process.</p>
<p>The major differences between <code>hipcc</code> and <code>amdclang++</code> are listed below:</p>
<p>::::{table} Differences between <code>hipcc</code> and <code>amdclang++</code>
:name: hipcc-vs-amdclang
| *                                  | <strong><code>hipcc</code></strong>                                                                                                              | <strong><code>amdclang++</code></strong> |
|:----------------------------------:|:------------------------------------------------------------------------------------------------------------------------:|:----------------:|
| Compiling HIP source files         | Treats all source files as HIP language source files                                                                     | Enables the HIP language support for files with the <code>.hip</code> extension or through the <code>-x hip</code> compiler option |
| Detecting GPU architecture         | Auto-detects the GPUs available on the system and generates code for those devices when no GPU architecture is specified | Has AMD GCN gfx803 as the default GPU architecture. The <code>--offload-arch</code> compiler option may be used to target other GPU architectures |
| Finding a HIP installation         | Finds the HIP installation based on its own location and its knowledge about the ROCm directory structure                | First looks for HIP under the same parent directory as its own LLVM directory and then falls back on <code>/opt/rocm</code>. Users can use the <code>--rocm-path</code> option to instruct the compiler to use HIP from the specified ROCm installation. |
| Linking to the HIP runtime library | Is configured to automatically link to the HIP runtime from the detected HIP installation                                | Requires the <code>--hip-link</code> flag to be specified to link to the HIP runtime. Alternatively, users can use the <code>-l&lt;dir&gt; -lamdhip64</code> option to link to a HIP runtime library. |
| Device function inlining           | Inlines all GPU device functions, which provide greater performance and compatibility for codes that contain file scoped or device function scoped <code>__shared__</code> variables. However, it may increase compile time. | Relies on inlining heuristics to control inlining. Users experiencing performance or compilation issues with code using file scoped or device function scoped <code>__shared__</code> variables could try <code>-mllvm -amdgpu-early-inline-all=true -mllvm -amdgpu-function-calls=false</code> to work around the issue. There are plans to address these issues with future compiler improvements. |
| Source code location               | <a href="https://github.com/ROCm-Developer-Tools/HIPCC">https://github.com/ROCm-Developer-Tools/HIPCC</a>                                                                          | <a href="https://github.com/RadeonOpenCompute/llvm-project">https://github.com/RadeonOpenCompute/llvm-project</a> |
::::</p>
<h2>Compiler Options and Features</h2>
<p>This chapter discusses compiler options and features.</p>
<h3>AMD GPU Compilation</h3>
<p>This section outlines commonly used compiler flags for <code>hipcc</code> and <code>amdclang++</code>.
:::{option} -x hip
  Compiles the source file as a HIP program.
:::</p>
<p>:::{option} -fopenmp
  Enables the OpenMP support.
:::</p>
<p>:::{option} -fopenmp-targets=<gpu>
  Enables the OpenMP target offload support of the specified GPU architecture.</p>
<p>:gpu: The GPU architecture. E.g. gfx908.
:::</p>
<p>:::{option} --gpu-max-threads-per-block=<value>:
  Sets the default limit of threads per block. Also referred to as the launch bounds.</p>
<p>:value: The default maximum amount of threads per block.
:::</p>
<p>:::{option} -munsafe-fp-atomics
  Enables unsafe floating point atomic instructions (AMDGPU only).
:::</p>
<p>:::{option} -ffast-math
  Allows aggressive, lossy floating-point optimizations.
:::</p>
<p>:::{option} -mwavefrontsize64, -mno-wavefrontsize64
  Sets wavefront size to be 64 or 32 on RDNA architectures.
:::</p>
<p>:::{option} -mcumode
  Switches between CU and WGP modes on RDNA architectures.
:::</p>
<p>:::{option} --offload-arch=<gpu>
  HIP offloading target ID. May be specified more than once.</p>
<p>:gpu: The a device architecture followed by target ID features
    delimited by a colon. Each target ID feature is a predefined
    string followed by a plus or minus sign (e.g. <code>gfx908:xnack+:sramecc-</code>).
:::</p>
<p>:::{option} -g
  Generates source-level debug information.
:::</p>
<p>:::{option} -fgpu-rdc, -fno-gpu-rdc
  Generates relocatable device code, also known as separate compilation mode.
:::</p>
<h3>AMD Optimizations for Zen Architectures</h3>
<p>The CPU compiler optimizations described in this chapter originate from the AMD
Optimizing C/C++ Compiler (AOCC) compiler. They are available in ROCmCC if the
optional <code>rocm-llvm-alt</code> package is installed. The user’s interaction with the
compiler does not change once <code>rocm-llvm-alt</code> is installed. The user should use
the same compiler entry point, provided AMD provides high-performance compiler
optimizations for Zen-based processors in AOCC.</p>
<p>For more information, refer to
<a href="https://www.amd.com/en/developer/aocc.html">https://www.amd.com/en/developer/aocc.html</a>.</p>
<h4><code>-famd-opt</code></h4>
<p>Enables a default set of AMD proprietary optimizations for the AMD Zen CPU
architectures.</p>
<p><code>-fno-amd-opt</code> disables the AMD proprietary optimizations.</p>
<p>The <code>-famd-opt</code> flag is useful when a user wants to build with the proprietary
optimization compiler and not have to depend on setting any of the other
proprietary optimization flags.</p>
<p>:::{note}
<code>-famd-opt</code> can be used in addition to the other proprietary CPU optimization
flags. The table of optimizations below implicitly enables the invocation of the
AMD proprietary optimizations compiler, whereas the <code>-famd-opt</code> flag requires
this to be handled explicitly.
:::</p>
<h4><code>-fstruct-layout=[1,2,3,4,5,6,7]</code></h4>
<p>Analyzes the whole program to determine if the structures in the code can be
peeled and the pointer or integer fields in the structure can be compressed. If
feasible, this optimization transforms the code to enable these improvements.
This transformation is likely to improve cache utilization and memory bandwidth.
It is expected to improve the scalability of programs executed on multiple cores.</p>
<p>This is effective only under <code>-flto</code>, as the whole program analysis is required
to perform this optimization. Users can choose different levels of
aggressiveness with which this optimization can be applied to the application,
with 1 being the least aggressive and 7 being the most aggressive level.</p>
<p>:::{table} -fstruct-layout Values and Their Effects
| <code>-fstruct-layout</code> value | Structure peeling | Pointer size after selective compression of self-referential pointers in structures, wherever safe | Type of structure fields eligible for compression | Whether compression performed under safety check |
| ----------- | ----------- | ----------- | ----------- | ----------- |
| 1 | Enabled | NA | NA | NA |
| 2 | Enabled | 32-bit | NA | NA |
| 3 | Enabled | 16-bit | NA | NA |
| 4 | Enabled | 32-bit | Integer | Yes |
| 5 | Enabled | 16-bit | Integer | Yes |
| 6 | Enabled | 32-bit | 64-bit signed int or unsigned int. Users must ensure that the values assigned to 64-bit signed int fields are in range -(2^31 - 1) to +(2^31 - 1) and 64-bit unsigned int fields are in the range 0 to +(2^31 - 1). Otherwise, you may obtain incorrect results. | No. Users must ensure the safety based on the program compiled. |
| 7 | Enabled | 16-bit | 64-bit signed int or unsigned int. Users must ensure that the values assigned to 64-bit signed int fields are in range -(2^31 - 1) to +(2^31 - 1) and 64-bit unsigned int fields are in the range 0 to +(2^31 - 1). Otherwise, you may obtain incorrect results. | No. Users must ensure the safety based on the program compiled. |
:::</p>
<h4><code>-fitodcalls</code></h4>
<p>Promotes indirect-to-direct calls by placing conditional calls. Application or
benchmarks that have a small and deterministic set of target functions for
function pointers passed as call parameters benefit from this optimization.
Indirect-to-direct call promotion transforms the code to use all possible
determined targets under runtime checks and falls back to the original code for
all the other cases. Runtime checks are introduced by the compiler for each of
these possible function pointer targets followed by direct calls to the targets.</p>
<p>This is a link time optimization, which is invoked as <code>-flto -fitodcalls</code></p>
<h4><code>-fitodcallsbyclone</code></h4>
<p>Performs value specialization for functions with function pointers passed as an
argument. It does this specialization by generating a clone of the function. The
cloning of the function happens in the call chain as needed, to allow conversion
of indirect function call to direct call.</p>
<p>This complements <code>-fitodcalls</code> optimization and is also a link time
optimization, which is invoked as <code>-flto -fitodcallsbyclone</code>.</p>
<h4><code>-fremap-arrays</code></h4>
<p>Transforms the data layout of a single dimensional array to provide better cache
locality. This optimization is effective only under <code>-flto</code>, as the whole program
needs to be analyzed to perform this optimization, which can be invoked as
<code>-flto -fremap-arrays</code>.</p>
<h4><code>-finline-aggressive</code></h4>
<p>Enables improved inlining capability through better heuristics. This
optimization is more effective when used with <code>-flto</code>, as the whole program
analysis is required to perform this optimization, which can be invoked as
<code>-flto -finline-aggressive</code>.</p>
<h4><code>-fnt-store (non-temporal store)</code></h4>
<p>Generates a non-temporal store instruction for array accesses in a loop with a
large trip count.</p>
<h4><code>-fnt-store=aggressive</code></h4>
<p>This is an experimental option to generate non-temporal store instruction for
array accesses in a loop, whose iteration count cannot be determined at compile
time. In this case, the compiler assumes the iteration count to be huge.</p>
<h4>Optimizations Through Driver <code>-mllvm &lt;options&gt;</code></h4>
<p>The following optimization options must be invoked through driver
<code>-mllvm &lt;options&gt;</code>:</p>
<h5><code>-enable-partial-unswitch</code></h5>
<p>Enables partial loop unswitching, which is an enhancement to the existing loop
unswitching optimization in LLVM. Partial loop unswitching hoists a condition
inside a loop from a path for which the execution condition remains invariant,
whereas the original loop unswitching works for a condition that is completely
loop invariant. The condition inside the loop gets hoisted out from the
invariant path, and the original loop is retained for the path where the
condition is variant.</p>
<h5><code>-aggressive-loop-unswitch</code></h5>
<p>Experimental option that enables aggressive loop unswitching heuristic
(including <code>-enable-partial-unswitch</code>) based on the usage of the branch
conditional values. Loop unswitching leads to code bloat. Code bloat can be
minimized if the hoisted condition is executed more often. This heuristic
prioritizes the conditions based on the number of times they are used within the
loop. The heuristic can be controlled with the following options:</p>
<ul>
<li><code>-unswitch-identical-branches-min-count=&lt;n&gt;</code></li>
<li>Enables unswitching of a loop with respect to a branch conditional value
    (B), where B appears in at least <code>&lt;n&gt;</code> compares in the loop. This option is
    enabled with <code>-aggressive-loop-unswitch</code>. The default value is 3.</li>
</ul>
<p><strong>Usage:</strong> <code>-mllvm -aggressive-loop-unswitch -mllvm -unswitch-identical-branches-min-count=&lt;n&gt;</code></p>
<p>Where, <code>n</code> is a positive integer and lower value of <code>&lt;n&gt;</code> facilitates more
  unswitching.
- <code>-unswitch-identical-branches-max-count=&lt;n&gt;</code>
  - Enables unswitching of a loop with respect to a branch conditional value
    (B), where B appears in at most <code>&lt;n&gt;</code> compares in the loop. This option is
    enabled with <code>-aggressive-loop-unswitch</code>. The default value is 6.</p>
<p><strong>Usage:</strong> <code>-mllvm -aggressive-loop-unswitch -mllvm -unswitch-identical-branches-max-count=&lt;n&gt;</code></p>
<p>Where, <code>n</code> is a positive integer and higher value of <code>&lt;n&gt;</code> facilitates more unswitching.</p>
<p>:::{note}
   These options may facilitate more unswitching under some workloads. Since
   loop-unswitching inherently leads to code bloat, facilitating more
   unswitching may significantly increase the code size. Hence, it may also lead
   to longer compilation times.
   :::</p>
<h5><code>-enable-strided-vectorization</code></h5>
<p>Enables strided memory vectorization as an enhancement to the interleaved
vectorization framework present in LLVM. It enables the effective use of gather
and scatter kind of instruction patterns. This flag must be used along with the
interleave vectorization flag.</p>
<h5><code>-enable-epilog-vectorization</code></h5>
<p>Enables vectorization of epilog-iterations as an enhancement to existing
vectorization framework. This enables generation of an additional epilog vector
loop version for the remainder iterations of the original vector loop. The
vector size or factor of the original loop should be large enough to allow an
effective epilog vectorization of the remaining iterations. This optimization
takes place only when the original vector loop is vectorized with a vector width
or factor of 16. This vectorization width of 16 may be overwritten by
<code>-min-width-epilog-vectorization</code> command-line option.</p>
<h5><code>-enable-redundant-movs</code></h5>
<p>Removes any redundant <code>mov</code> operations including redundant loads from memory and
stores to memory. This can be invoked using
<code>-Wl,-plugin-opt=-enable-redundant-movs</code>.</p>
<h5><code>-merge-constant</code></h5>
<p>Attempts to promote frequently occurring constants to registers. The aim is to
reduce the size of the instruction encoding for instructions using constants and
obtain a performance improvement.</p>
<h5><code>-function-specialize</code></h5>
<p>Optimizes the functions with compile time constant formal arguments.</p>
<h5><code>-lv-function-specialization</code></h5>
<p>Generates specialized function versions when the loops inside function are
vectorizable and the arguments are not aliased with each other.</p>
<h5><code>-enable-vectorize-compares</code></h5>
<p>Enables vectorization on certain loops with conditional breaks assuming the
memory accesses are safely bound within the page boundary.</p>
<h5><code>-inline-recursion=[1,2,3,4]</code></h5>
<p>Enables inlining for recursive functions based on heuristics where the
aggressiveness of heuristics increases with the level (1-4). The default level
is 2. Higher levels may lead to code bloat due to expansion of recursive
functions at call sites.</p>
<p>:::{table} -inline-recursion Level and Their Effects
| <code>-inline-recursion</code> <strong>value</strong> | <strong>Inline depth of heuristics used to enable inlining for recursive functions</strong> |
|:-----------------------------:|:------------------------------------------------------------------------------:|
| 1                             | 1                                                                              |
| 2                             | 1                                                                              |
| 3                             | 1                                                                              |
| 4                             | 10                                                                             |
:::</p>
<p>This is more effective with <code>-flto</code> as the whole program needs to be analyzed to
perform this optimization, which can be invoked as
<code>-flto -inline-recursion=[1,2,3,4]</code>.</p>
<h5><code>-reduce-array-computations=[1,2,3]</code></h5>
<p>Performs array data flow analysis and optimizes the unused array computations.</p>
<p>:::{table} -reduce-array-computations Values and Their Effects
| -reduce-array-computations value | Array elements eligible for elimination of computations |
| -------------------------------- | --------------------------- |
| 1                                | Unused                      |
| 2                                | Zero valued                 |
| 3                                | Both unused and zero valued |
:::</p>
<p>This optimization is effective with <code>-flto</code> as the whole program needs to be
analyzed to perform this optimization, which can be invoked as
<code>-flto -reduce-array-computations=[1,2,3]</code>.</p>
<h5><code>-global-vectorize-slp={true,false}</code></h5>
<p>Vectorizes the straight-line code inside a basic block with data reordering
vector operations. This option is set to <strong>true</strong> by default.</p>
<h5><code>-region-vectorize</code></h5>
<p>Experimental flag for enabling vectorization on certain loops with complex
control flow, which the normal vectorizer cannot handle.</p>
<p>This optimization is effective with <code>-flto</code> as the whole program needs to be
analyzed to perform this optimization, which can be invoked as
<code>-flto -region-vectorize</code>.</p>
<h5><code>-enable-x86-prefetching</code></h5>
<p>Enables the generation of x86 prefetch instruction for the memory references
inside a loop or inside an innermost loop of a loop nest to prefetch the second
dimension of multidimensional array/memory references in the innermost loop of a
loop nest. This is an experimental pass; its profitability is being improved.</p>
<h5><code>-suppress-fmas</code></h5>
<p>Identifies the reduction patterns on FMA and suppresses the FMA generation, as
it is not profitable on the reduction patterns.</p>
<h5><code>-enable-icm-vrp</code></h5>
<p>Enables estimation of the virtual register pressure before performing loop
invariant code motion. This estimation is used to control the number of loop
invariants that will be hoisted during the loop invariant code motion.</p>
<h5><code>-loop-splitting</code></h5>
<p>Enables splitting of loops into multiple loops to eliminate the branches, which
compare the loop induction with an invariant or constant expression. This option
is enabled under <code>-O3</code> by default. To disable this optimization, use
<code>-loop-splitting=false</code>.</p>
<h5><code>-enable-ipo-loop-split</code></h5>
<p>Enables splitting of loops into multiple loops to eliminate the branches, which
compares the loop induction with a constant expression. This constant expression
can be derived through inter-procedural analysis. This option is enabled under
<code>-O3</code> by default. To disable this optimization, use
<code>-enable-ipo-loop-split=false</code>.</p>
<h5><code>-compute-interchange-order</code></h5>
<p>Enables heuristic for finding the best possible interchange order for a loop
nest. To enable this option, use <code>-enable-loopinterchange</code>. This option is set
to <strong>false</strong> by default.</p>
<p><strong>Usage:</strong></p>
<p><code>bash
-mllvm -enable-loopinterchange -mllvm -compute-interchange-order</code></p>
<h5><code>-convert-pow-exp-to-int={true,false}</code></h5>
<p>Converts the call to floating point exponent version of pow to its integer
exponent version if the floating-point exponent can be converted to integer.
This option is set to <strong>true</strong> by default.</p>
<h5><code>-do-lock-reordering={none,normal,aggressive}</code></h5>
<p>Reorders the control predicates in increasing order of complexity from outer
predicate to inner when it is safe. The <strong>normal</strong> mode reorders simple
expressions, while the <strong>aggressive</strong> mode reorders predicates involving
function calls if no side effects are determined. This option is set to
<strong>normal</strong> by default.</p>
<h5><code>-fuse-tile-inner-loop</code></h5>
<p>Enables fusion of adjacent tiled loops as a part of loop tiling transformation.
This option is set to false by default.</p>
<h5><code>-Hz,1,0x1 [Fortran]</code></h5>
<p>Helps to preserve array index information for array access expressions which get
linearized in the compiler front end. The preserved information is used by the
compiler optimization phase in performing optimizations such as loop
transformations. It is recommended that any user who is using optimizations
such as loop transformations and other optimizations requiring de-linearized
index expressions should use the Hz option. This option has no impact on any
other aspects of the Flang front end.</p>
<h3>Inline ASM Statements</h3>
<p>Inline assembly (ASM) statements allow a developer to include assembly
instructions directly in either host or device code. While the ROCm compiler
supports ASM statements, their use is not recommended for the following reasons:</p>
<ul>
<li>The compiler's ability to produce both correct code and to optimize
  surrounding code is impeded.</li>
<li>The compiler does not parse the content of the ASM statements and so
  cannot "see" its contents.</li>
<li>The compiler must make conservative assumptions in an effort to retain
  correctness.</li>
<li>The conservative assumptions may yield code that, on the whole, is less
  performant compared to code without ASM statements. It is possible that a
  syntactically correct ASM statement may cause incorrect runtime behavior.</li>
<li>ASM statements are often ASIC-specific; code containing them is less portable
  and adds a maintenance burden to the developer if different ASICs are
  targeted.</li>
<li>Writing correct ASM statements is often difficult; we strongly recommend
  thorough testing of any use of ASM statements.</li>
</ul>
<p>:::{note}
For developers who choose to include ASM statements in the code, AMD is
interested in understanding the use case and appreciates feedback at
<a href="https://github.com/RadeonOpenCompute/ROCm/issues">https://github.com/RadeonOpenCompute/ROCm/issues</a>
:::</p>
<h3>Miscellaneous OpenMP Compiler Features</h3>
<p>This section discusses features that have been added or enhanced in the OpenMP
compiler.</p>
<h4>Offload-arch Tool</h4>
<p>An LLVM library and tool that is used to query the execution capability of the
current system as well as to query requirements of a binary file. It is used by
OpenMP device runtime to ensure compatibility of an image with the current
system while loading it. It is compatible with target ID support and multi-image
fat binary support.</p>
<p><strong>Usage:</strong></p>
<p><code>bash
offload-arch [Options] [Optional lookup-value]</code></p>
<p>When used without an option, offload-arch prints the value of the first offload
arch found in the underlying system. This can be used by various clang
front ends. For example, to compile for OpenMP offloading on your current system,
invoke clang with the following command:</p>
<p><code>bash
clang -fopenmp -fopenmp-targets=`offload-arch` foo.c</code></p>
<p>If an optional lookup-value is specified, offload-arch will check if the value
is either a valid offload-arch or a codename and look up requested additional
information.</p>
<p>The following command provides all the information for offload-arch gfx906:</p>
<p><code>bash
offload-arch gfx906 -v</code></p>
<p>The options are listed below:</p>
<p>:::{program} offload-arch
:::{option} -h
  Prints the help message.
:::</p>
<p>:::{option} -a
  Prints values for all devices. Do not stop at the first device found.
:::</p>
<p>:::{option} -m
  Prints device code name (often found in <code>pci.ids</code> file).
:::</p>
<p>:::{option} -n
  Prints numeric <code>pci-id</code>.
:::</p>
<p>:::{option} -t
   Prints clang offload triple to use for the offload arch.
:::</p>
<p>:::{option} -v
  Verbose. Implies: <code>-a -m -n -t</code>. For: all devices, prints codename, numeric value, and triple.
:::</p>
<p>:::{option} -f <file>
   Prints offload requirements including offload-arch for each compiled offload image built into an application binary file.
:::</p>
<p>:::{option} -c
  Prints offload capabilities of the underlying system. This option is used by the language runtime to select an image when multiple images are available. A capability must exist for each requirement of the selected image.
:::</p>
<p>There are symbolic link aliases <code>amdgpu-offload-arch</code> and <code>nvidia-arch</code> for
<code>offload-arch</code>. These aliases return 1 if no AMD GCN GPU or CUDA GPU is found.
These aliases are useful in determining whether architecture-specific tests
should be run or to conditionally load architecture-specific software.</p>
<h4>Command-Line Simplification Using <code>offload-arch</code> Flag</h4>
<p>Legacy mechanism of specifying offloading target for OpenMP involves using three
flags, <code>-fopenmp-targets</code>, <code>-Xopenmp-target</code>, and <code>-march</code>. The first two flags
take a target triple (like <code>amdgcn-amd-amdhsa</code> or <code>nvptx64-nvidia-cuda</code>), while
the last flag takes device name (like <code>gfx908</code> or <code>sm_70</code>) as input.
Alternatively, users of ROCmCC compiler can use the flag <code>—offload-arch</code> for a
combined effect of the above three flags.</p>
<p><strong>Example:</strong></p>
<p>```bash</p>
<h1>Legacy mechanism</h1>
<p>clang -fopenmp -target x86_64-linux-gnu \
-fopenmp-targets=amdgcn-amd-amdhsa -Xopenmp-target=amdgcn-amd-amdhsa \
-march=gfx906 helloworld.c -o helloworld
```</p>
<p><strong>Example:</strong></p>
<p>```bash</p>
<h1>Using offload-arch flag</h1>
<p>clang -fopenmp -target x86_64-linux-gnu \
--offload-arch=gfx906 helloworld.c -o helloworld.
```</p>
<p>To ensure backward compatibility, both styles are supported. This option is
compatible with target ID support and multi-image fat binaries.</p>
<h4>Target ID Support for OpenMP</h4>
<p>The ROCmCC compiler supports specification of target features along with the GPU
name while specifying a target offload device in the command line, using
<code>-march</code> or <code>--offload-arch</code> options. The compiled image in such cases is
specialized for a given configuration of device and target features (target ID).</p>
<p><strong>Example:</strong></p>
<p>```bash</p>
<h1>compiling for a gfx908 device with XNACK paging support turned ON</h1>
<p>clang -fopenmp -target x86_64-linux-gnu \
-fopenmp-targets=amdgcn-amd-amdhsa -Xopenmp-target=amdgcn-amd-amdhsa \
-march=gfx908:xnack+ helloworld.c -o helloworld
```</p>
<p><strong>Example:</strong></p>
<p>```bash</p>
<h1>compiling for a gfx908 device with SRAMECC support turned OFF</h1>
<p>clang -fopenmp -target x86_64-linux-gnu \
-fopenmp-targets=amdgcn-amd-amdhsa -Xopenmp-target=amdgcn-amd-amdhsa \
-march=gfx908:sramecc- helloworld.c -o helloworld
```</p>
<p><strong>Example:</strong></p>
<p>```bash</p>
<h1>compiling for a gfx908 device with SRAMECC support turned ON and XNACK paging support turned OFF</h1>
<p>clang -fopenmp -target x86_64-linux-gnu \
-fopenmp-targets=amdgcn-amd-amdhsa -Xopenmp-target=amdgcn-amd-amdhsa \
-march=gfx908:sramecc+:xnack- helloworld.c -o helloworld
```</p>
<p>The target ID specified on the command line is passed to the clang driver using
<code>target-feature</code> flag, to the LLVM optimizer and back end using <code>-mattr</code> flag, and
to linker using <code>-plugin-opt=-mattr</code> flag. This feature is compatible with
offload-arch command-line option and multi-image binaries for multiple
architectures.</p>
<h4>Multi-image Fat Binary for OpenMP</h4>
<p>The ROCmCC compiler is enhanced to generate binaries that can contain
heterogenous images. This heterogeneity could be in terms of:</p>
<ul>
<li>Images of different architectures, like AMD GCN and NVPTX</li>
<li>Images of same architectures but for different GPUs, like gfx906 and gfx908</li>
<li>Images of same architecture and same GPU but for different target features,
  like <code>gfx908:xnack+</code> and <code>gfx908:xnack-</code></li>
</ul>
<p>An appropriate image is selected by the OpenMP device runtime for execution
depending on the capability of the current system. This feature is compatible
with target ID support and offload-arch command-line options and uses
offload-arch tool to determine capability of the current system.</p>
<p><strong>Example:</strong></p>
<p><code>bash
clang -fopenmp -target x86_64-linux-gnu \
-fopenmp-targets=amdgcn-amd-amdhsa,amdgcn-amd-amdhsa \
-Xopenmp-target=amdgcn-amd-amdhsa -march=gfx906 \
-Xopenmp-target=amdgcn-amd-amdhsa -march=gfx908 \
helloworld.c -o helloworld</code></p>
<p><strong>Example:</strong></p>
<p><code>bash
clang -fopenmp -target x86_64-linux-gnu \
--offload-arch=gfx906 \
--offload-arch=gfx908 \
helloworld.c -o helloworld</code></p>
<p><strong>Example:</strong></p>
<p><code>bash
clang -fopenmp -target x86_64-linux-gnu \
-fopenmp-targets=amdgcn-amd-amdhsa,amdgcn-amd-amdhsa,amdgcn-amd-amdhsa,amdgcn-amd-amdhsa \
-Xopenmp-target=amdgcn-amd-amdhsa -march=gfx908:sramecc+:xnack+ \
-Xopenmp-target=amdgcn-amd-amdhsa -march=gfx908:sramecc-:xnack+ \
-Xopenmp-target=amdgcn-amd-amdhsa -march=gfx908:sramecc+:xnack- \
-Xopenmp-target=amdgcn-amd-amdhsa -march=gfx908:sramecc-:xnack- \
helloworld.c -o helloworld</code></p>
<p>The ROCmCC compiler creates an instance of toolchain for each unique combination
of target triple and the target GPU (along with the associated target features).
<code>clang-offload-wrapper</code> tool is modified to insert a new structure
<code>__tgt_image_info</code> along with each image in the binary. Device runtime is also
modified to query this structure to identify a compatible image based on the
capability of the current system.</p>
<h4>Unified Shared Memory (USM)</h4>
<p>The following OpenMP pragma is available on MI200, and it must be executed with
<code>xnack+</code> support.</p>
<p><code>cpp
omp requires unified_shared_memory</code></p>
<p>For more details on USM refer to the {ref}<code>openmp_usm</code> section of the OpenMP
Guide.</p>
