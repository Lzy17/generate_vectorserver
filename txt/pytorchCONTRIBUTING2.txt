<h2>Contributing to PyTorch</h2>
<p>Thank you for your interest in contributing to PyTorch! Before you begin writing code, it is important
that you share your intention to contribute with the team, based on the type of contribution:</p>
<ol>
<li>You want to propose a new feature and implement it.<ul>
<li>Post about your intended feature in an <a href="https://github.com/pytorch/pytorch/issues">issue</a>,
and we shall discuss the design and implementation. Once we agree that the plan looks good,
go ahead and implement it.</li>
</ul>
</li>
<li>You want to implement a feature or bug-fix for an outstanding issue.<ul>
<li>Search for your issue in the <a href="https://github.com/pytorch/pytorch/issues">PyTorch issue list</a>.</li>
<li>Pick an issue and comment that you'd like to work on the feature or bug-fix.</li>
<li>If you need more context on a particular issue, please ask and we shall provide.</li>
</ul>
</li>
</ol>
<p>Once you implement and test your feature or bug-fix, please submit a Pull Request to
https://github.com/pytorch/pytorch.</p>
<p>This document covers some of the more technical aspects of contributing
to PyTorch.  For more non-technical guidance about how to contribute to
PyTorch, see the <a href="docs/source/community/contribution_guide.rst">Contributing Guide</a>.</p>
<h2>Developing PyTorch</h2>
<p>Follow the instructions for <a href="https://github.com/pytorch/pytorch#from-source">installing PyTorch from source</a>. If you get stuck when developing PyTorch on your machine, check out the <a href="#tips-and-debugging">tips and debugging</a> section below for common solutions.</p>
<h3>Tips and Debugging</h3>
<ul>
<li>
<p>If you want to have no-op incremental rebuilds (which are fast), see <a href="#make-no-op-build-fast">Make no-op build fast</a> below.</p>
</li>
<li>
<p>When installing with <code>python setup.py develop</code> (in contrast to <code>python setup.py install</code>) Python runtime will use
  the current local source-tree when importing <code>torch</code> package. (This is done by creating <a href="https://wiki.python.org/moin/PythonPackagingTerminology#egg-link"><code>.egg-link</code></a> file in <code>site-packages</code> folder)
  This way you do not need to repeatedly install after modifying Python files (<code>.py</code>).
  However, you would need to reinstall if you modify Python interface (<code>.pyi</code>, <code>.pyi.in</code>) or
   non-Python files (<code>.cpp</code>, <code>.cc</code>, <code>.cu</code>, <code>.h</code>, ...).</p>
</li>
</ul>
<p>One way to avoid running <code>python setup.py develop</code> every time one makes a change to C++/CUDA/ObjectiveC files on Linux/Mac,
  is to create a symbolic link from <code>build</code> folder to <code>torch/lib</code>, for example, by issuing following:
  <code>bash
   pushd torch/lib; sh -c "ln -sf ../../build/lib/libtorch_cpu.* ."; popd</code>
   Afterwards rebuilding a library (for example to rebuild <code>libtorch_cpu.so</code> issue <code>ninja torch_cpu</code> from <code>build</code> folder),
   would be sufficient to make change visible in <code>torch</code> package.</p>
<p>To reinstall, first uninstall all existing PyTorch installs. You may need to run <code>pip
  uninstall torch</code> multiple times. You'll know <code>torch</code> is fully
  uninstalled when you see <code>WARNING: Skipping torch as it is not
  installed</code>. (You should only have to <code>pip uninstall</code> a few times, but
  you can always <code>uninstall</code> with <code>timeout</code> or in a loop if you're feeling
  lazy.)</p>
<p><code>bash
  conda uninstall pytorch -y
  yes | pip uninstall torch</code></p>
<p>Next run <code>python setup.py clean</code>. After that, you can install in <code>develop</code> mode again.</p>
<ul>
<li>
<p>If a commit is simple and doesn't affect any code (keep in mind that some docstrings contain code
  that is used in tests), you can add <code>[skip ci]</code> (case sensitive) somewhere in your commit message to
  <a href="https://github.blog/changelog/2021-02-08-github-actions-skip-pull-request-and-push-workflows-with-skip-ci/">skip all build / test steps</a>.
  Note that changing the pull request body or title on GitHub itself has no effect.</p>
</li>
<li>
<p>If you run into errors when running <code>python setup.py develop</code>, here are some debugging steps:</p>
</li>
<li>Run <code>printf '#include &lt;stdio.h&gt;\nint main() { printf("Hello World");}'|clang -x c -; ./a.out</code> to make sure
  your CMake works and can compile this simple Hello World program without errors.</li>
<li>Nuke your <code>build</code> directory. The <code>setup.py</code> script compiles binaries into the <code>build</code> folder and caches many
  details along the way, which saves time the next time you build. If you're running into issues, you can always
  <code>rm -rf build</code> from the toplevel <code>pytorch</code> directory and start over.</li>
<li>If you have made edits to the PyTorch repo, commit any change you'd like to keep and clean the repo with the
  following commands (note that clean <em>really</em> removes all untracked files and changes.):
      <code>bash
      git submodule deinit -f .
      git clean -xdf
      python setup.py clean
      git submodule update --init --recursive # very important to sync the submodules
      python setup.py develop                 # then try running the command again</code></li>
<li>
<p>The main step within <code>python setup.py develop</code> is running <code>make</code> from the <code>build</code> directory. If you want to
    experiment with some environment variables, you can pass them into the command:
      <code>bash
      ENV_KEY1=ENV_VAL1[, ENV_KEY2=ENV_VAL2]* python setup.py develop</code></p>
</li>
<li>
<p>If you run into issue running <code>git submodule update --init --recursive</code>. Please try the following:</p>
</li>
<li>
<p>If you encounter an error such as
    <code>error: Submodule 'third_party/pybind11' could not be updated</code>
    check whether your Git local or global config file contains any <code>submodule.*</code> settings. If yes, remove them and try again.
    (please reference <a href="https://git-scm.com/docs/git-config#Documentation/git-config.txt-submoduleltnamegturl">this doc</a> for more info).</p>
</li>
<li>
<p>If you encounter an error such as
    <code>fatal: unable to access 'https://github.com/pybind11/pybind11.git': could not load PEM client certificate ...</code>
    this is likely that you are using HTTP proxying and the certificate expired. To check if the certificate is valid, run
    <code>git config --global --list</code> and search for config like <code>http.proxysslcert=&lt;cert_file&gt;</code>. Then check certificate valid date by running
    <code>bash
    openssl x509 -noout -in &lt;cert_file&gt; -dates</code></p>
</li>
<li>
<p>If you encounter an error that some third_party modules are not checked out correctly, such as
    <code>Could not find .../pytorch/third_party/pybind11/CMakeLists.txt</code>
    remove any <code>submodule.*</code> settings in your local git config (<code>.git/config</code> of your pytorch repo) and try again.</p>
</li>
<li>If you're a Windows contributor, please check out <a href="https://github.com/pytorch/pytorch/wiki/Best-Practices-to-Edit-and-Compile-Pytorch-Source-Code-On-Windows">Best Practices</a>.</li>
<li>For help with any part of the contributing process, please donâ€™t hesitate to utilize our Zoom office hours! See details <a href="https://github.com/pytorch/pytorch/wiki/Dev-Infra-Office-Hours">here</a></li>
</ul>
<h2>Nightly Checkout &amp; Pull</h2>
<p>The <code>tools/nightly.py</code> script is provided to ease pure Python development of
PyTorch. This uses <code>conda</code> and <code>git</code> to check out the nightly development
version of PyTorch and installs pre-built binaries into the current repository.
This is like a development or editable install, but without needing the ability
to compile any C++ code.</p>
<p>You can use this script to check out a new nightly branch with the following:</p>
<p><code>bash
./tools/nightly.py checkout -b my-nightly-branch
conda activate pytorch-deps</code></p>
<p>Or if you would like to re-use an existing conda environment, you can pass in
the regular environment parameters (<code>--name</code> or <code>--prefix</code>):</p>
<p><code>bash
./tools/nightly.py checkout -b my-nightly-branch -n my-env
conda activate my-env</code></p>
<p>You can also use this tool to pull the nightly commits into the current branch:</p>
<p><code>bash
./tools/nightly.py pull -n my-env
conda activate my-env</code></p>
<p>Pulling will reinstall the PyTorch dependencies as well as the nightly binaries
into the repo directory.</p>

<h2>Unit testing</h2>
<h3>Python Unit Testing</h3>
<p><strong>Prerequisites</strong>:
The following packages should be installed with either <code>conda</code> or <code>pip</code>:
- <code>expecttest</code> and <code>hypothesis</code> - required to run tests
- <code>mypy</code> - recommended for linting
- <code>pytest</code> - recommended to run tests more selectively</p>
<p>All PyTorch test suites are located in the <code>test</code> folder and start with
<code>test_</code>. Run the entire test
suite with</p>
<p><code>bash
python test/run_test.py</code></p>
<p>or run individual test suites using the command <code>python test/FILENAME.py</code>,
where <code>FILENAME</code> represents the file containing the test suite you wish
to run.</p>
<p>For example, to run all the TorchScript JIT tests (located at
<code>test/test_jit.py</code>), you would run:</p>
<p><code>bash
python test/test_jit.py</code></p>
<p>You can narrow down what you're testing even further by specifying the
name of an individual test with <code>TESTCLASSNAME.TESTNAME</code>. Here,
<code>TESTNAME</code> is the name of the test you want to run, and <code>TESTCLASSNAME</code>
is the name of the class in which it is defined.</p>
<p>Going off the above example, let's say you want to run
<code>test_Sequential</code>, which is defined as part of the <code>TestJit</code> class
in <code>test/test_jit.py</code>. Your command would be:</p>
<p><code>bash
python test/test_jit.py TestJit.test_Sequential</code></p>
<p><strong>Weird note:</strong> In our CI (Continuous Integration) jobs, we actually run the tests from the <code>test</code> folder and <strong>not</strong> the root of the repo, since there are various dependencies we set up for CI that expects the tests to be run from the test folder. As such, there may be some inconsistencies between local testing and CI testing--if you observe an inconsistency, please <a href="https://github.com/pytorch/pytorch/issues/new/choose">file an issue</a>.</p>
<h3>Better local unit tests with <code>pytest</code></h3>
<p>We don't officially support <code>pytest</code>, but it works well with our
<code>unittest</code> tests and offers a number of useful features for local
developing. Install it via <code>pip install pytest</code>.</p>
<p>If you want to just run tests that contain a specific substring, you can
use the <code>-k</code> flag:</p>
<p><code>bash
pytest test/test_nn.py -k Loss -v</code></p>
<p>The above is an example of testing a change to all Loss functions: this
command runs tests such as <code>TestNN.test_BCELoss</code> and
<code>TestNN.test_MSELoss</code> and can be useful to save keystrokes.</p>
<h3>Local linting</h3>
<p>Install all prerequisites by running</p>
<p><code>bash
make setup_lint</code></p>
<p>You can now run the same linting steps that are used in CI locally via <code>make</code>:</p>
<p><code>bash
make lint</code></p>
<p>Learn more about the linter on the <a href="https://github.com/pytorch/pytorch/wiki/lintrunner">lintrunner wiki page</a></p>
<h4>Running <code>mypy</code></h4>
<p><code>mypy</code> is an optional static type checker for Python. We have multiple <code>mypy</code>
configs for the PyTorch codebase that are automatically validated against whenever the linter is run.</p>
<p>See <a href="https://github.com/pytorch/pytorch/wiki/Guide-for-adding-type-annotations-to-PyTorch">Guide for adding type annotations to
PyTorch</a>
for more information on how to set up <code>mypy</code> and tackle type annotation
tasks.</p>
<h3>C++ Unit Testing</h3>
<p>PyTorch offers a series of tests located in the <code>test/cpp</code> folder.
These tests are written in C++ and use the Google Test testing framework.
After compiling PyTorch from source, the test runner binaries will be
written to the <code>build/bin</code> folder. The command to run one of these tests
is <code>./build/bin/FILENAME --gtest_filter=TESTSUITE.TESTNAME</code>, where
<code>TESTNAME</code> is the name of the test you'd like to run and <code>TESTSUITE</code> is
the suite that test is defined in.</p>
<p>For example, if you wanted to run the test <code>MayContainAlias</code>, which
is part of the test suite <code>ContainerAliasingTest</code> in the file
<code>test/cpp/jit/test_alias_analysis.cpp</code>, the command would be:</p>
<p><code>bash
./build/bin/test_jit --gtest_filter=ContainerAliasingTest.MayContainAlias</code></p>
<h3>Run Specific CI Jobs</h3>
<p>You can generate a commit that limits the CI to only run a specific job by using
<code>tools/testing/explicit_ci_jobs.py</code> like so:</p>
<p>```bash</p>
<h1>--job: specify one or more times to filter to a specific job + its dependencies</h1>
<h1>--filter-gha: specify github actions workflows to keep</h1>
<h1>--make-commit: commit CI changes to git with a message explaining the change</h1>
<p>python tools/testing/explicit_ci_jobs.py --job binary_linux_manywheel_3_6m_cpu_devtoolset7_nightly_test --filter-gha '<em>generated</em>gcc5.4*' --make-commit</p>
<h1>Make your changes</h1>
<p>ghstack submit
```</p>
<p><strong>NB</strong>: It is not recommended to use this workflow unless you are also using
<a href="https://github.com/ezyang/ghstack"><code>ghstack</code></a>. It creates a large commit that is
of very low signal to reviewers.</p>
<h2>Merging your Change</h2>
<p>If you know the right people or team that should approve your PR (and you have the required permissions to do so), add them to the Reviewers list.</p>
<p>If not, leave the Reviewers section empty. Our triage squad will review your PR, add a module label, and assign it to the appropriate reviewer in a couple business days.  The reviewer will then look at your PR and respond.</p>
<p>Occasionally, things might fall through the cracks (sorry!). In case your PR either doesn't get assigned to a reviewer or doesn't get any response from the reviewer for 4 business days, please leave comment on the PR (mentioning the reviewer if one has been assigned). That'll get it nudged back onto people's radar.</p>
<p>If that still doesn't help, come see us during <a href="https://github.com/pytorch/pytorch/wiki/Contact-Pytorch-Dev-Infra-Office">our office hours</a></p>
<p>Once your PR is approved, you can merge it in by entering a comment with the content <code>@pytorchmergebot merge</code> (<a href="https://github.com/pytorch/pytorch/wiki/Bot-commands">what's this bot?</a>)</p>
<h2>Writing documentation</h2>
<p>So you want to write some documentation and don't know where to start?
PyTorch has two main types of documentation:
- <strong>User facing documentation</strong>:
These are the docs that you see over at <a href="https://pytorch.org/docs">our docs website</a>.
- <strong>Developer facing documentation</strong>:
Developer facing documentation is spread around our READMEs in our codebase and in
the <a href="https://pytorch.org/wiki">PyTorch Developer Wiki</a>.
If you're interested in adding new developer docs, please read this <a href="https://github.com/pytorch/pytorch/wiki/Where-or-how-should-I-add-documentation">page on the wiki</a> on our best practices for where to put it.</p>
<p>The rest of this section is about user-facing documentation.</p>
<p>PyTorch uses <a href="https://www.sphinx-doc.org/en/master/usage/extensions/example_google.html">Google style</a>
for formatting docstrings. Each line inside a docstrings block must be limited to 80 characters so that it fits into Jupyter documentation popups.</p>
<h3>Docstring type formatting</h3>
<p>In addition to the standard Google Style docstring formatting rules, the following guidelines should be followed for docstring types (docstring types are the type information contained in the round brackets after the variable name):</p>
<ul>
<li>
<p>The "<code>Callable</code>", "<code>Any</code>", "<code>Iterable</code>", "<code>Iterator</code>", "<code>Generator</code>" types should have their first letter capitalized.</p>
</li>
<li>
<p>The "<code>list</code>" and "<code>tuple</code>" types should be completely lowercase.</p>
</li>
<li>
<p>Types should not be made plural. For example: <code>tuple of int</code> should be used instead of <code>tuple of ints</code>.</p>
</li>
<li>
<p>The only acceptable delimiter words for types are <code>or</code> and <code>of</code>. No other non-type words should be used other than <code>optional</code>.</p>
</li>
<li>
<p>The word <code>optional</code> should only be used after the types, and it is only used if the user does not have to specify a value for the variable. Default values are listed after the variable description. Example:</p>
<p><code>my_var (int, optional): Variable description. Default: 1</code></p>
</li>
<li>
<p>Basic Python types should match their type name so that the <a href="https://www.sphinx-doc.org/en/master/usage/extensions/intersphinx.html">Intersphinx</a> extension can correctly identify them. For example:</p>
<ul>
<li>Use <code>str</code> instead of <code>string</code>.</li>
<li>Use <code>bool</code> instead of <code>boolean</code>.</li>
<li>Use <code>dict</code> instead of <code>dictionary</code>.</li>
</ul>
</li>
<li>
<p>Square brackets should be used for the dictionary type. For example:</p>
<p><code>my_var (dict[str, int]): Variable description.</code></p>
</li>
<li>
<p>If a variable has two different possible types, then the word <code>or</code> should be used without a comma. Otherwise variables with 3 or more types should use commas to separate the types. Example:</p>
<p><code>x (type1 or type2): Variable description.
y (type1, type2, or type3): Variable description.</code></p>
</li>
</ul>

<h2>Profiling with <code>py-spy</code></h2>
<p>Evaluating the performance impact of code changes in PyTorch can be complicated,
particularly if code changes happen in compiled code. One simple way to profile
both Python and C++ code in PyTorch is to use
<a href="https://github.com/benfred/py-spy"><code>py-spy</code></a>, a sampling profiler for Python
that has the ability to profile native code and Python code in the same session.</p>
<p><code>py-spy</code> can be installed via <code>pip</code>:</p>
<p><code>bash
pip install py-spy</code></p>
<p>To use <code>py-spy</code>, first write a Python test script that exercises the
functionality you would like to profile. For example, this script profiles
<code>torch.add</code>:</p>
<p>```python
import torch</p>
<p>t1 = torch.tensor([[1, 1], [1, 1.]])
t2 = torch.tensor([[0, 0], [0, 0.]])</p>
<p>for _ in range(1000000):
    torch.add(t1, t2)
```</p>
<p>Since the <code>torch.add</code> operation happens in microseconds, we repeat it a large
number of times to get good statistics. The most straightforward way to use
<code>py-spy</code> with such a script is to generate a <a href="http://www.brendangregg.com/flamegraphs.html">flame
graph</a>:</p>
<p><code>bash
py-spy record -o profile.svg --native -- python test_tensor_tensor_add.py</code></p>
<p>This will output a file named <code>profile.svg</code> containing a flame graph you can
view in a web browser or SVG viewer. Individual stack frame entries in the graph
can be selected interactively with your mouse to zoom in on a particular part of
the program execution timeline. The <code>--native</code> command-line option tells
<code>py-spy</code> to record stack frame entries for PyTorch C++ code. To get line numbers
for C++ code it may be necessary to compile PyTorch in debug mode by prepending
your <code>setup.py develop</code> call to compile PyTorch with <code>DEBUG=1</code>. Depending on
your operating system it may also be necessary to run <code>py-spy</code> with root
privileges.</p>
<p><code>py-spy</code> can also work in an <code>htop</code>-like "live profiling" mode and can be
tweaked to adjust the stack sampling rate, see the <code>py-spy</code> readme for more
details.</p>

<h2>C++ development tips</h2>
<p>If you are working on the C++ code, there are a few important things that you
will want to keep in mind:</p>
<ol>
<li>How to rebuild only the code you are working on.</li>
<li>How to make rebuilds in the absence of changes go faster.</li>
</ol>
<h3>Build only what you need</h3>
<p><code>python setup.py build</code> will build everything by default, but sometimes you are
only interested in a specific component.</p>
<ul>
<li>Working on a test binary? Run <code>(cd build &amp;&amp; ninja bin/test_binary_name)</code> to
  rebuild only that test binary (without rerunning cmake). (Replace <code>ninja</code> with
  <code>make</code> if you don't have ninja installed).</li>
<li>Don't need Caffe2?  Pass <code>BUILD_CAFFE2=0</code> to disable Caffe2 build.</li>
</ul>
<p>On the initial build, you can also speed things up with the environment
variables <code>DEBUG</code>, <code>USE_DISTRIBUTED</code>, <code>USE_MKLDNN</code>, <code>USE_CUDA</code>, <code>BUILD_TEST</code>, <code>USE_FBGEMM</code>, <code>USE_NNPACK</code> and <code>USE_QNNPACK</code>.</p>
<ul>
<li><code>DEBUG=1</code> will enable debug builds (-g -O0)</li>
<li><code>REL_WITH_DEB_INFO=1</code> will enable debug symbols with optimizations (-g -O3)</li>
<li><code>USE_DISTRIBUTED=0</code> will disable distributed (c10d, gloo, mpi, etc.) build.</li>
<li><code>USE_MKLDNN=0</code> will disable using MKL-DNN.</li>
<li><code>USE_CUDA=0</code> will disable compiling CUDA (in case you are developing on something not CUDA related), to save compile time.</li>
<li><code>BUILD_TEST=0</code> will disable building C++ test binaries.</li>
<li><code>USE_FBGEMM=0</code> will disable using FBGEMM (quantized 8-bit server operators).</li>
<li><code>USE_NNPACK=0</code> will disable compiling with NNPACK.</li>
<li><code>USE_QNNPACK=0</code> will disable QNNPACK build (quantized 8-bit operators).</li>
<li><code>USE_XNNPACK=0</code> will disable compiling with XNNPACK.</li>
</ul>
<p>For example:</p>
<p><code>bash
DEBUG=1 USE_DISTRIBUTED=0 USE_MKLDNN=0 USE_CUDA=0 BUILD_TEST=0 USE_FBGEMM=0 USE_NNPACK=0 USE_QNNPACK=0 USE_XNNPACK=0 python setup.py develop</code></p>
<p>For subsequent builds (i.e., when <code>build/CMakeCache.txt</code> exists), the build
options passed for the first time will persist; please run <code>ccmake build/</code>, run
<code>cmake-gui build/</code>, or directly edit <code>build/CMakeCache.txt</code> to adapt build
options.</p>
<h3>Code completion and IDE support</h3>
<p>When using <code>python setup.py develop</code>, PyTorch will generate
a <code>compile_commands.json</code> file that can be used by many editors
to provide command completion and error highlighting for PyTorch's
C++ code. You need to <code>pip install ninja</code> to generate accurate

<h3>Make no-op build fast</h3>
<h4>Use Ninja</h4>
<p>By default, cmake will use its Makefile generator to generate your build
system.  You can get faster builds if you install the ninja build system
with <code>pip install ninja</code>.  If PyTorch was already built, you will need
to run <code>python setup.py clean</code> once after installing ninja for builds to
succeed.</p>
<h4>Use CCache</h4>
<p>Even when dependencies are tracked with file modification, there are many
situations where files get rebuilt when a previous compilation was exactly the
same. Using ccache in a situation like this is a real time-saver.</p>
<p>Before building pytorch, install ccache from your package manager of choice:</p>
<p><code>bash
conda install ccache -c conda-forge
sudo apt install ccache
sudo yum install ccache
brew install ccache</code></p>
<p>You may also find the default cache size in ccache is too small to be useful.
The cache sizes can be increased from the command line:</p>
<p>```bash</p>
<h1>config: cache dir is ~/.ccache, conf file ~/.ccache/ccache.conf</h1>
<h1>max size of cache</h1>
<p>ccache -M 25Gi  # -M 0 for unlimited</p>
<h1>unlimited number of files</h1>
<p>ccache -F 0
```</p>
<p>To check this is working, do two clean builds of pytorch in a row. The second
build should be substantially and noticeably faster than the first build. If
this doesn't seem to be the case, check the <code>CMAKE_&lt;LANG&gt;_COMPILER_LAUNCHER</code>
rules in <code>build/CMakeCache.txt</code>, where <code>&lt;LANG&gt;</code> is <code>C</code>, <code>CXX</code> and <code>CUDA</code>.
Each of these 3 variables should contain ccache, e.g.</p>
<p><code>//CXX compiler launcher
CMAKE_CXX_COMPILER_LAUNCHER:STRING=/usr/bin/ccache</code></p>
<p>If not, you can define these variables on the command line before invoking <code>setup.py</code>.</p>
<p><code>bash
export CMAKE_C_COMPILER_LAUNCHER=ccache
export CMAKE_CXX_COMPILER_LAUNCHER=ccache
export CMAKE_CUDA_COMPILER_LAUNCHER=ccache
python setup.py develop</code></p>
<h4>Use a faster linker</h4>
<p>If you are editing a single file and rebuilding in a tight loop, the time spent
linking will dominate. The system linker available in most Linux distributions
(GNU <code>ld</code>) is quite slow. Use a faster linker, like <a href="https://lld.llvm.org/">lld</a>.</p>
<p>People on Mac, follow <a href="https://stackoverflow.com/questions/42730345/how-to-install-llvm-for-mac">this guide</a> instead.</p>
<p>The easiest way to use <code>lld</code> this is download the
<a href="http://releases.llvm.org/download.html#8.0.0">latest LLVM binaries</a> and run:</p>
<p><code>bash
ln -s /path/to/downloaded/ld.lld /usr/local/bin/ld</code></p>
<h4>Use pre-compiled headers</h4>
<p>Sometimes there's no way of getting around rebuilding lots of files, for example
editing <code>native_functions.yaml</code> usually means 1000+ files being rebuilt. If
you're using CMake newer than 3.16, you can enable pre-compiled headers by
setting <code>USE_PRECOMPILED_HEADERS=1</code> either on first setup, or in the
<code>CMakeCache.txt</code> file.</p>
<p><code>sh
USE_PRECOMPILED_HEADERS=1 python setup.py develop</code></p>
<p>This adds a build step where the compiler takes <code>&lt;ATen/ATen.h&gt;</code> and essentially
dumps it's internal AST to a file so the compiler can avoid repeating itself for
every <code>.cpp</code> file.</p>
<p>One caveat is that when enabled, this header gets included in every file by default.
Which may change what code is legal, for example:
- internal functions can never alias existing names in <code>&lt;ATen/ATen.h&gt;</code>
- names in <code>&lt;ATen/ATen.h&gt;</code> will work even if you don't explicitly include it.</p>
<h4>Workaround for header dependency bug in nvcc</h4>
<p>If re-building without modifying any files results in several CUDA files being
re-compiled, you may be running into an <code>nvcc</code> bug where header dependencies are
not converted to absolute paths before reporting it to the build system. This
makes <code>ninja</code> think one of the header files has been deleted, so it runs the
build again.</p>
<p>A compiler-wrapper to fix this is provided in <code>tools/nvcc_fix_deps.py</code>. You can use
this as a compiler launcher, similar to <code>ccache</code>
<code>bash
export CMAKE_CUDA_COMPILER_LAUNCHER="python;`pwd`/tools/nvcc_fix_deps.py;ccache"
python setup.py develop</code></p>

<h2>CUDA development tips</h2>
<p>If you are working on the CUDA code, here are some useful CUDA debugging tips:</p>
<ol>
<li><code>CUDA_DEVICE_DEBUG=1</code> will enable CUDA device function debug symbols (<code>-g -G</code>).
    This will be particularly helpful in debugging device code. However, it will
    slow down the build process for about 50% (compared to only <code>DEBUG=1</code>), so use wisely.</li>
<li><code>cuda-gdb</code> and <code>cuda-memcheck</code> are your best CUDA debugging friends. Unlike<code>gdb</code>,
   <code>cuda-gdb</code> can display actual values in a CUDA tensor (rather than all zeros).</li>
<li>CUDA supports a lot of C++11/14 features such as, <code>std::numeric_limits</code>, <code>std::nextafter</code>,
   <code>std::tuple</code> etc. in device code. Many of such features are possible because of the
   <a href="https://docs.nvidia.com/cuda/cuda-c-programming-guide/index.html#constexpr-functions">--expt-relaxed-constexpr</a>
   nvcc flag. There is a known <a href="https://github.com/ROCm-Developer-Tools/HIP/issues/374">issue</a>
   that ROCm errors out on device code, which uses such stl functions.</li>
<li>A good performance metric for a CUDA kernel is the
   <a href="https://devblogs.nvidia.com/how-implement-performance-metrics-cuda-cc/">Effective Memory Bandwidth</a>.
   It is useful for you to measure this metric whenever you are writing/optimizing a CUDA
   kernel. Following script shows how we can measure the effective bandwidth of CUDA <code>uniform_</code>
   kernel.
   ```python
   import torch
   from torch.utils.benchmark import Timer
   size = 128*512
   nrep = 100
   nbytes_read_write = 4 # this is number of bytes read + written by a kernel. Change this to fit your kernel.</li>
</ol>
<p>for i in range(10):
       a=torch.empty(size).cuda().uniform_()
       torch.cuda.synchronize()
       out = a.uniform_()
       torch.cuda.synchronize()
       t = Timer(stmt="a.uniform_()", globals=globals())
       res = t.blocked_autorange()
       timec = res.median
       print("uniform, size, elements", size, "forward", timec, "bandwidth (GB/s)", size<em>(nbytes_read_write)</em>1e-9/timec)
       size *=2
   ```</p>
<p>See more cuda development tips <a href="https://github.com/pytorch/pytorch/wiki/CUDA-basics">here</a></p>
<h2>Windows development tips</h2>
<p>For building from source on Windows, consult
<a href="https://pytorch.org/docs/stable/notes/windows.html">our documentation</a> on it.</p>
<p>Occasionally, you will write a patch which works on Linux, but fails CI on Windows.
There are a few aspects in which MSVC (the Windows compiler toolchain we use) is stricter
than Linux, which are worth keeping in mind when fixing these problems.</p>
<ol>
<li>Symbols are NOT exported by default on Windows; instead, you have to explicitly
   mark a symbol as exported/imported in a header file with <code>__declspec(dllexport)</code> /
   <code>__declspec(dllimport)</code>. We have codified this pattern into a set of macros
   which follow the convention <code>*_API</code>, e.g., <code>TORCH_API</code> inside Caffe2, Aten and Torch.
   (Every separate shared library needs a unique macro name, because symbol visibility
   is on a per shared library basis. See c10/macros/Macros.h for more details.)</li>
</ol>
<p>The upshot is if you see an "unresolved external" error in your Windows build, this
   is probably because you forgot to mark a function with <code>*_API</code>. However, there is
   one important counterexample to this principle: if you want a <em>templated</em> function
   to be instantiated at the call site, do NOT mark it with <code>*_API</code> (if you do mark it,
   you'll have to explicitly instantiate all of the specializations used by the call
   sites.)</p>
<ol>
<li>
<p>If you link against a library, this does not make its dependencies transitively
   visible. You must explicitly specify a link dependency against every library whose
   symbols you use. (This is different from Linux where in most environments,
   transitive dependencies can be used to fulfill unresolved symbols.)</p>
</li>
<li>
<p>If you have a Windows box (we have a few on EC2 which you can request access to) and
   you want to run the build, the easiest way is to just run <code>.ci/pytorch/win-build.sh</code>.
   If you need to rebuild, run <code>REBUILD=1 .ci/pytorch/win-build.sh</code> (this will avoid
   blowing away your Conda environment.)</p>
</li>
</ol>
<p>Even if you don't know anything about MSVC, you can use cmake to build simple programs on
Windows; this can be helpful if you want to learn more about some peculiar linking behavior
by reproducing it on a small example. Here's a simple example cmake file that defines
two dynamic libraries, one linking with the other:</p>
<p>```CMake
project(myproject CXX)
set(CMAKE_CXX_STANDARD 14)
add_library(foo SHARED foo.cpp)
add_library(bar SHARED bar.cpp)</p>
<h1>NB: don't forget to __declspec(dllexport) at least one symbol from foo,</h1>
<h1>otherwise foo.lib will not be created.</h1>
<p>target_link_libraries(bar PUBLIC foo)
```</p>
<p>You can build it with:</p>
<p><code>bash
mkdir build
cd build
cmake ..
cmake --build .</code></p>
<h3>Known MSVC (and MSVC with NVCC) bugs</h3>
<p>The PyTorch codebase sometimes likes to use exciting C++ features, and
these exciting features lead to exciting bugs in Windows compilers.
To add insult to injury, the error messages will often not tell you
which line of code actually induced the erroring template instantiation.</p>
<p>We've found the most effective way to debug these problems is to
carefully read over diffs, keeping in mind known bugs in MSVC/NVCC.
Here are a few well known pitfalls and workarounds:</p>
<ul>
<li>
<p>This is not actually a bug per se, but in general, code generated by MSVC
  is more sensitive to memory errors; you may have written some code
  that does a use-after-free or stack overflows; on Linux the code
  might work, but on Windows your program will crash. ASAN may not
  catch all of these problems: stay vigilant to the possibility that
  your crash is due to a real memory problem.</p>
</li>
<li>
<p>(NVCC) <code>c10::optional</code> does not work when used from device code. Don't use
  it from kernels. Upstream issue: https://github.com/akrzemi1/Optional/issues/58
  and our local issue #10329.</p>
</li>
<li>
<p><code>constexpr</code> generally works less well on MSVC.</p>
</li>
<li>
<p>The idiom <code>static_assert(f() == f())</code> to test if <code>f</code> is constexpr
    does not work; you'll get "error C2131: expression did not evaluate
    to a constant". Don't use these asserts on Windows.
    (Example: <code>c10/util/intrusive_ptr.h</code>)</p>
</li>
<li>
<p>(NVCC) Code you access inside a <code>static_assert</code> will eagerly be
  evaluated as if it were device code, and so you might get an error
  that the code is "not accessible".</p>
</li>
</ul>
<p><code>cpp
class A {
  static A singleton_;
  static constexpr inline A* singleton() {
    return &amp;singleton_;
  }
};
static_assert(std::is_same(A*, decltype(A::singleton()))::value, "hmm");</code></p>
<ul>
<li>
<p>The compiler will run out of heap space if you attempt to compile files that
  are too large. Splitting such files into separate files helps.
  (Example: <code>THTensorMath</code>, <code>THTensorMoreMath</code>, <code>THTensorEvenMoreMath</code>.)</p>
</li>
<li>
<p>MSVC's preprocessor (but not the standard compiler) has a bug
  where it incorrectly tokenizes raw string literals, ending when it sees a <code>"</code>.
  This causes preprocessor tokens inside the literal like an<code>#endif</code>  to be incorrectly
  treated as preprocessor directives. See https://godbolt.org/z/eVTIJq as an example.</p>
</li>
<li>
<p>Either MSVC or the Windows headers have a PURE macro defined and will replace
  any occurrences of the PURE token in code with an empty string. This is why
  we have AliasAnalysisKind::PURE_FUNCTION and not AliasAnalysisKind::PURE.
  The same is likely true for other identifiers that we just didn't try to use yet.</p>
</li>
</ul>

<h2>Building PyTorch with ASAN</h2>
<p><a href="https://github.com/google/sanitizers/wiki/AddressSanitizer">ASAN</a> is very
useful for debugging memory errors in C++. We run it in CI, but here's how to
get the same thing to run on your local machine.</p>
<p>First, install LLVM 8. The easiest way is to get <a href="http://releases.llvm.org/download.html#8.0.0">prebuilt
binaries</a> and extract them to
folder (later called <code>$LLVM_ROOT</code>).</p>
<p>Then set up the appropriate scripts. You can put this in your <code>.bashrc</code>:</p>
<p>```bash
LLVM_ROOT=<wherever your llvm install is>
PYTORCH_ROOT=<wherever your pytorch checkout is></p>
<p>LIBASAN_RT="$LLVM_ROOT/lib/clang/8.0.0/lib/linux/libclang_rt.asan-x86_64.so"
build_with_asan()
{
  LD_PRELOAD=${LIBASAN_RT} \
  CC="$LLVM_ROOT/bin/clang" \
  CXX="$LLVM_ROOT/bin/clang++" \
  LDSHARED="clang --shared" \
  LDFLAGS="-stdlib=libstdc++" \
  CFLAGS="-fsanitize=address -fno-sanitize-recover=all -shared-libasan -pthread" \
  CXX_FLAGS="-pthread" \
  USE_CUDA=0 USE_OPENMP=0 BUILD_CAFFE2_OPS=0 USE_DISTRIBUTED=0 DEBUG=1 \
  python setup.py develop
}</p>
<p>run_with_asan()
{
  LD_PRELOAD=${LIBASAN_RT} $@
}</p>
<h1>you can look at build-asan.sh to find the latest options the CI uses</h1>
<p>export ASAN_OPTIONS=detect_leaks=0:symbolize=1:strict_init_order=true
export UBSAN_OPTIONS=print_stacktrace=1:suppressions=$PYTORCH_ROOT/ubsan.supp
export ASAN_SYMBOLIZER_PATH=$LLVM_ROOT/bin/llvm-symbolizer
```</p>
<p>Then you can use the scripts like:</p>
<p><code>suo-devfair ~/pytorch â¯ build_with_asan
suo-devfair ~/pytorch â¯ run_with_asan python test/test_jit.py</code></p>
<h3>Getting <code>ccache</code> to work</h3>
<p>The scripts above specify the <code>clang</code> and <code>clang++</code> binaries directly, which
bypasses <code>ccache</code>. Here's how to get <code>ccache</code> to work:</p>
<ol>
<li>Make sure the ccache symlinks for <code>clang</code> and <code>clang++</code> are set up (see
   CONTRIBUTING.md)</li>
<li>Make sure <code>$LLVM_ROOT/bin</code> is available on your <code>$PATH</code>.</li>
<li>Change the <code>CC</code> and <code>CXX</code> variables in <code>build_with_asan()</code> to point
   directly to <code>clang</code> and <code>clang++</code>.</li>
</ol>
<h3>Why this stuff with <code>LD_PRELOAD</code> and <code>LIBASAN_RT</code>?</h3>
<p>The â€œstandardâ€ workflow for ASAN assumes you have a standalone binary:</p>
<ol>
<li>Recompile your binary with <code>-fsanitize=address</code>.</li>
<li>Run the binary, and ASAN will report whatever errors it find.</li>
</ol>
<p>Unfortunately, PyTorch is a distributed as a shared library that is loaded by
a third-party executable (Python). Itâ€™s too much of a hassle to recompile all
of Python every time we want to use ASAN. Luckily, the ASAN folks have a
workaround for cases like this:</p>
<ol>
<li>Recompile your library with <code>-fsanitize=address -shared-libasan</code>. The
   extra <code>-shared-libasan</code> tells the compiler to ask for the shared ASAN
   runtime library.</li>
<li>Use <code>LD_PRELOAD</code> to tell the dynamic linker to load the ASAN runtime
   library before anything else.</li>
</ol>
<p>More information can be found
<a href="https://github.com/google/sanitizers/wiki/AddressSanitizerAsDso">here</a>.</p>
<h3>Why LD_PRELOAD in the build function?</h3>
<p>We need <code>LD_PRELOAD</code> because there is a cmake check that ensures that a
simple program builds and runs. If we are building with ASAN as a shared
library, we need to <code>LD_PRELOAD</code> the runtime library, otherwise there will
dynamic linker errors and the check will fail.</p>
<p>We donâ€™t actually need either of these if we fix the cmake checks.</p>
<h3>Why no leak detection?</h3>
<p>Python leaks a lot of memory. Possibly we could configure a suppression file,
but we havenâ€™t gotten around to it.</p>

<h2>CI failure tips</h2>
<p>Once you submit a PR or push a new commit to a branch that is in
an active PR, CI jobs will be run automatically. Some of these may
fail and you will need to find out why, by looking at the logs.</p>
<p>Fairly often, a CI failure might be unrelated to your changes. You can
confirm by going to our <a href="https://hud.pytorch.org">HUD</a> and seeing if the CI job
is failing upstream already. In this case, you
can usually ignore the failure. See <a href="#which-commit-is-used-in-ci">the following
subsection</a> for more details.</p>
<p>Some failures might be related to specific hardware or environment
configurations. In this case, if you're a Meta employee, you can ssh into
the job's session to perform manual debugging following the instructions in
our <a href="https://github.com/pytorch/pytorch/wiki/Debugging-using-with-ssh-for-Github-Actions">CI wiki</a>.</p>
<h3>Which commit is used in CI?</h3>
<p>For CI run on <code>main</code>, this repository is checked out for a given <code>main</code>
commit, and CI is run on that commit (there isn't really any other choice).</p>
<p>For PRs, however, it's a bit more complicated. Consider this commit graph, where
<code>main</code> is at commit <code>A</code>, and the branch for PR #42 (just a placeholder) is at
commit <code>B</code>:</p>
<p><code>o---o---B (refs/pull/42/head)
      /         \
     /           C (refs/pull/42/merge)
    /           /
---o---o---o---A (merge-destination) - usually main</code></p>
<p>There are two possible choices for which commit to use:</p>
<ol>
<li>Checkout commit <code>B</code>, the head of the PR (manually committed by the PR
   author).</li>
<li>Checkout commit <code>C</code>, the hypothetical result of what would happen if the PR
   were merged into it's destination (usually <code>main</code>).</li>
</ol>
<p>For all practical purposes, most people can think of the commit being used as
commit <code>B</code> (choice <strong>1</strong>).</p>
<p>However, if workflow files (which govern CI behavior) were modified (either by your PR or since dev branch were created ) there's
a nuance to know about:
The workflow files themselves get taken from checkpoint <code>C</code>, the merger of your
PR and the <code>main</code> branch. But only the workflow files get taken from that merged
checkpoint. Everything else (tests, code, etc) all get taken directly from your
PR's commit (commit <code>B</code>). Please note, this scenario would never affect PRs authored by <code>ghstack</code> as they would not automatically ingest the updates from default branch.</p>
